# Nexora Feature Documentation

## Product Overview

Nexora is a simple, trusted, and location-aware OJT tracking system for practicum students, teachers, and school administrators. Its core purpose is to make student attendance logging seamless while ensuring that time records are valid through geolocation and photo verification.

Nexora removes unnecessary approval friction by using system-enforced validation. If a student is within the allowed workplace radius and completes the required verification steps, the log is accepted automatically.

## Product Vision

Build an OJT tracking platform that is:

* easy for students to use daily
* reliable for teachers and coordinators to monitor
* difficult to abuse or falsify
* clean, mobile-friendly, and fast

## Core Product Principles

1. Simplicity first
   Students should be able to complete attendance in a few taps.

2. Trust by system design
   The system should verify attendance automatically instead of relying on manual approval.

3. Low admin friction
   Teachers should monitor, review, and report, not approve every single entry.

4. Mobile-first experience
   Students will likely use phones while on-site, so workflows must be optimized for mobile.

5. Clear accountability
   Every attendance record should contain time, location, and proof.

## User Roles

### 1. Student

Primary user who logs attendance, submits daily activity, and tracks required hours.

### 2. Teacher / Coordinator / Admin

Monitors student attendance, checks logs, reviews compliance, manages workplace assignments, and generates reports.

## Main System Flow

### Student Flow

1. Student logs in.
2. Student opens dashboard.
3. System checks whether the student has an assigned workplace.
4. Student taps Time In.
5. System verifies geolocation against workplace radius.
6. System requires photo capture.
7. If valid, session starts.
8. Student taps Time Out at the end of duty.
9. System again verifies geolocation.
10. System requires another photo.
11. Student enters a short daily activity summary.
12. System calculates rendered hours and stores the record.

### Admin Flow

1. Admin logs in.
2. Admin views dashboard summary.
3. Admin monitors students, workplace assignments, attendance status, and history.
4. Admin reviews attendance records with time, location status, and photos.
5. Admin generates summaries and exportable reports.

## Feature Documentation

## 1. Authentication and Access Control

### Purpose

Secure access and separate the experience for students and admins.

### Features

* email and password sign in
* forgot password flow
* role-based redirection after login
* protected routes
* session persistence

### MVP Notes

* only two roles for MVP: student and admin
* registration can be restricted if school prefers admin-created accounts

* assigned supervisor or coordinator
* status: active, completed, inactive

### Admin Profile Fields

* full name
* email
* department
* role title

## 3. Workplace Assignment

### Purpose

Connect each student to the location where attendance is valid.

### Features

* assign student to a company or office
* save official workplace name
* save workplace address
* save latitude and longitude
* set allowed attendance radius in meters
* assign start and end dates

### Validation Rules

* student cannot time in if no workplace is assigned
* student cannot time in if current date is outside assignment period

## 4. Geolocation Validation

### Purpose

Ensure attendance actions only happen when the student is physically near the assigned workplace.

### Features

* capture current device location on time in
* capture current device location on time out
* compare captured coordinates with assigned workplace coordinates
* enforce radius rule
* block action if outside allowed range

### Recommended MVP Behavior

* default radius: 100 meters, configurable by admin
* show student a clear error when outside the allowed area
* show current validation status before button press when possible

### UX Messaging Examples

* You are within your assigned workplace. You may continue.
* You are too far from your assigned workplace to time in.
* Location access is required to continue.

## 5. Photo Verification

### Purpose

Provide visual proof tied to attendance events.

### Features

* require photo at time in
* require photo at time out
* upload and store image securely
* attach image URLs to attendance record

### MVP Rules

* no photo, no time in
* no photo, no time out
* images should be compressed for performance
* camera capture should be preferred on mobile

## 6. Time Tracking Engine

### Purpose

Capture duty sessions accurately and compute rendered hours automatically.

### Features

* start session on successful time in
* end session on successful time out
* compute total worked time
* display today’s status
* prevent invalid sequences

### Rules

* cannot time out before timing in
* cannot time in twice while session is active
* one active session at a time
* hours are auto-computed and not manually entered by student

### Optional Business Rules

* minimum duty duration threshold
* maximum daily hours limit
* grace period for minor GPS drift

## 7. Daily Activity Log

### Purpose

Add meaningful context to each attendance record.

### Features

* required short text field after time out
* optional character minimum
* viewable by admin in attendance history

### Suggested Prompt

What tasks or activities did you complete today?

### MVP Recommendation

Keep this very simple. One text area only.

## Smart Product Decisions for Nexora

### 1. No Manual Approval

This is one of the strongest product decisions for Nexora.

Why it is better:

* removes bottlenecks
* reduces teacher workload
* makes attendance immediate and seamless
* keeps the process objective

### 2. Geofence as the Main Gatekeeper

The system itself decides whether a log is valid.
That makes Nexora more trustworthy and easier to operate.

### 3. Photo Capture at Both Ends

This strengthens verification while still keeping the process simple.

### 4. Minimal Daily Log Input

Too many fields will hurt adoption. Keep the daily narrative short.

## Validation Status Design

For MVP, keep statuses simple:

* valid
* blocked
* incomplete

This is better than approve/reject because it reflects system outcome instead of admin judgment.

## Non-Functional Requirements

### Performance

* attendance actions should feel instant
* image upload should be optimized
* dashboard pages should load quickly

### Security

* students can only access their own records
* admins can access system-wide monitoring views
* storage access should be protected
* row-level security should be enforced in Supabase

### Reliability

* prevent duplicate submissions
* gracefully handle GPS permission denial
* gracefully handle photo upload failure

### Usability

* responsive on mobile and desktop
* clear language
* low cognitive load
